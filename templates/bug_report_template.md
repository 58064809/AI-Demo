# 🐞 Template Name

缺陷报告模板

---

## 🎯 Purpose

本模板用于统一缺陷记录与缺陷分析的交付结构，确保不同 Agent、不同场景下输出的 Bug 信息具备一致的可读性、可复查性和可执行性。

本模板不替代 `standards/bug_standard.md`，而是作为其交付载体使用。

---

## 📦 Default Structure

默认包含以下内容：

- Bug 标题
- 所属模块
- 前置条件
- 复现步骤
- 实际结果
- 预期结果
- 影响范围
- 严重程度建议
- 备注 / 补充说明

在需要更深入分析时，可补充：

- 根因分析
- 待确认项
- 修复建议
- 回归建议
- 补测建议

---

## ✅ Basic Template

### Bug 标题

【模块】+【触发场景】+【异常结果】

### 所属模块

请填写具体模块名称。

### 前置条件

请填写影响问题复现的必要条件，例如：

- 登录状态
- 角色类型
- 数据状态
- 环境版本
- 依赖条件

### 复现步骤

1.  
2.  
3.  
4.  
5.  

### 实际结果

请客观描述系统当前表现。

### 预期结果

请明确说明系统按规则应有的正确表现。

### 影响范围

请从以下维度说明：

- 用户影响
- 角色影响
- 功能影响
- 数据影响
- 业务影响

### 严重程度建议

- P0 / P1 / P2 / P3

### 备注 / 补充说明

请填写日志、截图、接口信息、环境信息、概率信息、临时绕过方案等补充内容。

## 📝 Writing Guidance

### Bug 标题

要求：

- 一眼能看懂问题是什么
- 能体现问题发生位置
- 不写成情绪化表达
- 不把猜测原因直接写成结论

---

### 复现步骤

要求：

- 顺序清晰
- 可直接执行
- 不跳关键步骤
- 不使用“按流程操作”“进行相关操作”这类模糊写法

---

### 实际结果

要求：

- 只描述事实
- 不夹带主观判断
- 尽量描述页面、接口、日志、数据中的可观察现象

---

### 预期结果

要求：

- 明确
- 可验证
- 可直接与实际结果对比
- 不写“符合预期”“正常处理”

---

### 影响范围

要求：

- 不只看当前点位
- 尽量说明用户、角色、模块、数据和业务影响
- 如无法完全确认，也应给出初步判断

---

### 严重程度建议

要求：

- 不拍脑袋
- 结合业务影响、数据影响、用户影响、可恢复性、可绕过性综合判断

---

## 🚫 Forbidden Usage

禁止以下写法：

- 只放截图，不写文字结构
- 只写“有问题”“页面异常”
- 标题写成“测试发现 bug”
- 实际结果写成“功能不对”
- 预期结果写成“应符合预期”
- 不写影响范围直接给严重程度
- 把猜测原因当成确定原因

---

## 🏁 Final Requirement

本模板的最终目标不是让 Bug“写满字段”，而是：

> 让缺陷信息能够被快速理解、准确复现、统一定级，并有效支持排查、修复、回归和质量决策。
