# 🧩 Template Name

API Mock 规格模板

---

## 🎯 Purpose

本模板用于统一 API Mock 规格说明的交付结构，确保不同 Agent、不同任务场景下输出的 Mock 设计内容具备一致的可读性、可实现性和可维护性。

本模板不替代 `skills/mock-server-design/SKILL.md`，而是作为 Mock 设计落地时的结构化交付载体使用。

---

## 📦 Default Structure

默认包含以下内容：

- Mock 目标
- 对应真实接口
- 适用场景
- 请求规格
- 返回规格
- 场景切换方式
- 状态 / 数据处理规则
- 已知限制
- 使用建议

根据场景不同，可用于：

- 联调占位 Mock
- 测试隔离 Mock
- 自动化测试 Mock
- 异常场景模拟 Mock
- 第三方依赖替代 Mock

---

### 7. 状态 / 数据处理规则

请说明当前 Mock 是否支持以下能力：

建议写清：

- 支持什么

---

### 8. 已知限制

请明确说明当前 Mock 的限制，例如：

- 仅支持部分字段

- 不支持复杂状态机
- 不支持异步链路
- 与真实接口仅部分对齐
- 仅适用于测试环境 / 联调环境
- 不能用于最终上线验证

---

## 📝 Writing Guidance

### Mock 目标

要求：

- 明确
- 单一
- 不空泛

禁止写法：

- 用于测试
- 方便联调
- 提供模拟数据

应尽量写清“为什么要做这个 Mock”。

---

### 对应真实接口

要求：

- 尽量说明真实接口路径和方法
- 尽量说明是否与真实契约一致
- 如不一致，应明确标出偏差点

---

### 请求 / 返回规格

要求：

- 结构化
- 可实现
- 可核对
- 不写模糊描述

---

### 场景切换方式

要求：

- 可操作
- 可重复
- 可配置
- 尽量不依赖手工改代码

---

### 已知限制

要求：

- 诚实
- 明确
- 可预期

不要把简化版 Mock 写得像完全等同真实系统。

---

## 🚫 Forbidden Usage

禁止以下写法：

- 只写“返回成功数据”
- 不写真实接口对应关系
- 不写场景切换方式
- 不写已知限制
- 不说明哪些行为是简化的
- 只有一段说明文字，没有结构化规格
- 把 Mock 当真实接口说明文档使用

---

## 🏁 Final Requirement

本模板的最终目标不是“写一份假接口说明”，而是：

> 让 Mock 服务设计具备明确目标、清晰契约、可控场景和边界说明，能够真正服务于联调、测试、自动化和依赖隔离，并减少误用和误判风险。
