# 🗂️ Template Name

测试计划模板

---

## 🎯 Purpose

本模板用于统一测试计划类内容的交付结构，确保不同 Agent、不同任务场景下输出的测试计划具备一致的可读性、可执行性和可追踪性。

本模板不替代 `standards/output_standard.md`、`standards/risk_standard.md` 或 `standards/review_standard.md`，而是作为测试活动启动前的统一规划载体使用。

---

## 📦 Default Structure

默认包含以下内容：

- 测试背景
- 测试目标
- 测试范围
- 测试策略
- 测试资源
- 测试环境
- 测试数据
- 风险与依赖
- 交付物
- 进度安排
- 准入 / 准出标准

根据场景不同，可用于：

- 版本测试计划
- 需求测试计划
- 回归测试计划
- 发布前测试计划
- 阶段性专项测试计划

---

## ✅ Standard Test Plan Template

### 1. 测试背景

请说明：

- 本次测试对应的需求、版本或项目背景
- 当前变更的业务目标
- 本次测试的触发原因（如版本发布、需求改版、缺陷修复、专项验证）

---

### 2. 测试目标

请说明：

- 本次测试希望验证什么
- 本次测试重点关注哪些质量目标
- 本次测试希望降低哪些风险

---

### 3. 测试范围

请说明：

- 本次覆盖哪些模块
- 本次覆盖哪些业务链路
- 本次重点验证哪些功能 / 页面 / 接口 / 状态流转
- 哪些内容明确不在本次测试范围内

---

### 4. 测试策略

请说明本次测试采用的主要策略，例如：

- 功能测试
- 接口测试
- UI 测试
- 回归测试
- 冒烟测试
- 性能 / 稳定性验证
- 权限测试
- 数据一致性验证

并说明：

- 测试重点
- 优先级策略
- 覆盖策略
- 是否有自动化配合

---

### 5. 测试资源

请说明：

- 参与角色
- 测试负责人
- 协作方
- 是否依赖开发 / 产品 / 运维 / 数据支持
- 当前资源是否充足

---

### 6. 测试环境

请说明：

- 测试环境名称
- 环境版本 / 构建信息
- 依赖服务是否可用
- 是否存在环境限制
- 是否存在联调依赖

---

### 7. 测试数据

请说明：

- 需要哪些测试数据
- 数据如何准备
- 是否依赖特定账号 / 角色 / 状态
- 是否需要造数 / Mock / 清理数据

---

### 8. 风险与依赖

请说明：

- 当前测试是否依赖未完成需求 / 未完成开发 / 外部系统 / 第三方服务
- 是否存在高风险区域
- 是否存在环境、数据、时间、资源上的约束
- 是否存在已知问题影响测试推进

---

### 9. 交付物

请说明本次测试计划对应的交付物，例如：

- 测试点
- 测试用例
- 缺陷报告
- 风险评估
- 测试报告
- 发布建议

---

### 10. 进度安排

请说明：

- 测试准备时间
- 测试执行时间
- 缺陷修复验证时间
- 回归时间
- 计划输出测试结论时间

如适合，也可用表格：

| 阶段 | 开始时间 | 结束时间 | 负责人 | 备注 |
|------|----------|----------|--------|------|
| 测试准备 |  |  |  |  |
| 用例设计 |  |  |  |  |
| 测试执行 |  |  |  |  |
| 回归验证 |  |  |  |  |
| 报告输出 |  |  |  |  |

---

### 11. 准入标准

请说明测试开始前必须满足的条件，例如：

- 需求已评审
- 开发提测完成
- 测试环境可用
- 核心依赖可联通
- 必要测试数据已准备
- 接口文档 / 原型 / 说明已齐备

---

### 12. 准出标准

请说明测试完成并进入下一阶段前应满足的条件，例如：

- 核心功能已完成测试
- P0 / P1 缺陷已关闭或有明确风险接受结论
- 高风险场景已覆盖
- 回归已完成
- 已输出测试结论与风险说明

## 📝 Writing Guidance

### 测试背景

要求：

- 简要说明本次测试为什么开始
- 不写过长背景故事
- 保持与本次测试直接相关

---

### 测试目标

要求：

- 目标清晰
- 可与后续测试结果对应
- 不写成空泛口号

禁止写法：

- 确保质量
- 提升稳定性
- 做好测试工作

---

### 测试范围

要求：

- 清楚说明“测什么”和“暂不测什么”
- 避免给人“已经全测完”的错觉
- 尤其在时间紧张时，要显式写出范围边界

---

### 测试策略

要求：

- 与当前任务匹配
- 能体现重点和优先级
- 不机械罗列测试类型

---

### 风险与依赖

要求：

- 尽量具体
- 说明风险点、依赖对象和可能影响
- 不写“存在一定风险”这种空话

---

### 准入 / 准出标准

要求：

- 明确
- 可判断
- 可执行
- 能作为测试启动和结束的依据

---

## 🚫 Forbidden Usage

禁止以下写法：

- 只有范围，没有目标
- 只有计划标题，没有执行约束
- 把所有内容都写成“待确认”
- 不区分范围内和范围外
- 不写风险与依赖
- 不写准入和准出标准
- 进度安排完全不可落地
- 全文只有大段描述，没有结构

---

## 🏁 Final Requirement

本模板的最终目标不是让测试计划“显得完整”，而是：

> 让测试活动在开始前就具备清晰目标、明确范围、合理策略和可执行约束，从而提升测试执行效率、降低遗漏风险，并为后续测试结论和质量决策打下基础。
