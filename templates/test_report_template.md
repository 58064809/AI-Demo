# 📊 Template Name

测试报告模板

---

## 🎯 Purpose

本模板用于统一测试报告类内容的交付结构，确保不同 Agent、不同任务场景下输出的测试结果具备一致的可读性、可比较性和可决策性。

本模板不替代 `standards/output_standard.md`、`standards/risk_standard.md` 或 `standards/review_standard.md`，而是作为测试结果汇总与测试结论交付的统一载体使用。

---

## 📦 Default Structure

默认包含以下内容：

- 报告概述
- 测试范围
- 测试环境
- 测试执行情况
- 发现的问题
- 风险评估
- 测试结论
- 后续建议

根据使用场景，可扩展为：

- 冒烟测试报告
- 回归测试报告
- 阶段测试报告
- 发布前测试结论
- 缺陷修复验证报告

---

## ✅ Standard Report Template

### 1. 报告概述

请说明：

- 本次测试的背景
- 测试目标
- 报告对应的版本 / 需求 / 发布范围

---

### 2. 测试范围

请说明：

- 本次覆盖了哪些模块
- 本次重点验证了哪些链路
- 哪些内容在本次范围内
- 哪些内容不在本次范围内

---

### 3. 测试环境

请说明：

- 测试环境名称
- 构建版本 / 包版本 / 分支信息
- 依赖服务是否齐备
- 是否有特殊环境说明

---

### 4. 测试执行情况

请说明：

- 执行了哪些类型测试
- 执行量级（如：测试点数、测试用例数、自动化执行数）
- 通过情况
- 失败情况
- 阻塞情况
- 未执行情况（如有）

如适合，也可用表格：

| 项目 | 数量 / 结果 |
|------|-------------|
| 测试点总数 |  |
| 测试用例总数 |  |
| 已执行 |  |
| 通过 |  |
| 失败 |  |
| 阻塞 |  |
| 未执行 |  |

---

### 5. 发现的问题

请说明：

- 本次测试发现了哪些关键问题
- 是否存在高优先级缺陷
- 是否存在未关闭问题
- 是否存在已知风险接受项

如适合，也可用表格：

| 问题 / 缺陷 | 等级 | 当前状态 | 影响范围 | 是否阻塞 |
|-------------|------|----------|----------|----------|

---

### 6. 风险评估

请说明：

- 当前版本还存在哪些主要风险
- 风险触发条件是什么
- 风险影响范围是什么
- 当前是否有缓解措施

如适合，可引用 `risk_assessment_template.md` 的结构展开。

---

### 7. 测试结论

结论建议尽量明确，可使用以下表述之一：

- 可通过
- 有条件通过
- 不建议通过

并说明：

- 为什么得出该结论
- 当前结论成立的前提
- 是否存在必须补充动作

---

### 8. 后续建议

请说明建议动作，例如：

- 继续修复哪些问题
- 补充哪些测试
- 增加哪些监控
- 是否建议灰度
- 是否建议延期
- 是否建议补回归 / 补性能验证

## ✅ Lightweight Smoke / Regression Report Template

适用于快速场景时，可使用简版：

### 测试目标

### 测试范围

### 执行结果

### 关键问题

### 当前风险

### 测试结论

### 后续建议

---

## 📝 Writing Guidance

### 报告概述

要求：

- 一句话说清测试背景
- 不写大段空泛背景介绍
- 不脱离本次测试目标

---

### 测试范围

要求：

- 范围要具体
- 清楚说明“测了什么”和“没测什么”
- 避免误导读者以为是全量覆盖

---

### 测试执行情况

要求：

- 尽量可量化
- 尽量能反映测试覆盖和执行结果
- 不只写“已完成测试”

---

### 问题与风险

要求：

- 区分已发现问题和潜在风险
- 问题偏事实
- 风险偏判断
- 不混在一起写成模糊描述

---

### 测试结论

要求：

- 必须明确
- 不能写“基本可以”
- 要说明依据和前提

---

### 后续建议

要求：

- 尽量具体
- 能指导动作
- 不写“持续关注”“后续优化”这种空泛建议

---

## 🚫 Forbidden Usage

禁止以下写法：

- 只写“测试完成”
- 只列问题，不给结论
- 只给结论，不给依据
- 不说明范围，却直接说“可以上线”
- 把已知缺陷和潜在风险混在一起
- 全文没有数量、没有状态、没有判断
- 用“基本正常”“整体没问题”这类模糊表述替代测试结论

---

## 🏁 Final Requirement

本模板的最终目标不是让测试报告“像一篇汇报材料”，而是：

> 让测试结果能够被快速理解、被管理决策使用、被后续动作承接，并真正服务于质量判断、风险评估和上线决策。
