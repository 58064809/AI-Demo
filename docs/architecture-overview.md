# 🏛️ Architecture Overview

---

## 🎯 Purpose

本文件用于说明本仓库的整体架构设计、分层目标、协作关系和演进思路。

它回答的问题包括：

- 这套企业级测试 Agent 仓库为什么这样设计
- 每一层分别解决什么问题
- 各层之间如何协作
- 为什么不能只靠一个超长 Prompt
- 后续应如何演进成更完整的测试能力体系

---

## 🧠 Overall Design Goal

本仓库的整体目标，不是简单堆积一批 Prompt，也不是只做一个临时性的 AI Demo，而是构建一套：

- 以资深测试工程师为唯一核心身份
- 面向企业真实测试场景
- 具备清晰职责边界
- 可扩展、可复用、可治理
- 能沉淀为长期团队资产

的测试 Agent 协作体系。

这意味着仓库中的每一类文件，都必须回答一个清晰的问题，而不是无序堆放内容。

---

## 🏗️ Layered Architecture

本仓库采用分层架构，整体可分为以下几层：

### 1. Identity Layer

对应文件：

- `AGENTS.md`

作用：

- 定义默认身份
- 定义总协作原则
- 定义加载优先级
- 定义跨层约束

这一层解决的问题是：

> 所有内容都必须站在同一个专业身份下工作，而不是今天像测试、明天像开发、后天像产品。

本仓库的唯一核心身份是：

> **资深测试工程师**

---

### 2. Role Layer

对应目录：

- `agents/`

作用：

- 将不同测试任务场景拆分为不同 Agent
- 定义每个 Agent 的职责边界、思考方式、输出要求和协作关系

这一层解决的问题是：

> 不同任务该由哪一种“测试视角分身”来处理。

例如：

- 需求分析代理
- 测试设计代理
- 接口测试代理
- UI 测试代理
- 缺陷分析代理
- 回归测试代理
- 性能与稳定性代理
- 测试架构代理

这些 Agent 不是不同职业，而是同一名资深测试工程师在不同任务场景下的专业分身。

---

### 3. Standard Layer

对应目录：

- `standards/`

作用：

- 定义所有 Agent 必须共享遵守的质量规范和输出规范
- 确保不同 Agent 输出风格一致、质量口径一致、结构标准一致

这一层解决的问题是：

> 即使不同 Agent 分工不同，最终输出也必须专业、一致、可复用。

例如：

- 输出规范
- 测试用例规范
- 缺陷规范
- 风险评估规范
- 评审规范

如果没有这一层，Agent 再多，也会因为标准不统一而迅速失控。

---

### 4. Skill Layer

对应目录：

- `skills/`

作用：

- 定义某类任务“具体应该怎么做”
- 把测试方法、处理步骤、分析技巧和常见检查点沉淀下来

这一层解决的问题是：

> Agent 负责“谁来做”，Skill 负责“具体怎么做”。

例如：

- 需求评审技能
- 测试用例设计技能
- 接口测试技能
- Bug Triage 技能
- Mock 设计技能
- 日志分析技能
- 自动化脚本技能
- 测试数据构建技能

Skill 不定义身份，它只定义方法。

---

### 5. Template Layer

对应目录：

- `templates/`

作用：

- 提供固定交付格式
- 统一交付结果结构
- 降低重复组织输出的成本

这一层解决的问题是：

> 当任务已经知道要输出什么时，用什么格式最适合沉淀和复用。

例如：

- 测试用例模板
- 缺陷报告模板
- 风险评估模板
- 评审清单模板
- 测试计划模板
- 测试报告模板

Template 不是规则，而是规则落地后的交付载体。

---

### 6. Reference Layer

对应目录：

- `references/`

作用：

- 沉淀长期背景资料
- 为 Agent 和 Skill 补充上下文

这一层解决的问题是：

> 当规则和方法已经确定后，还需要哪些背景知识来支撑更高质量判断。

例如：

- 术语表
- 系统背景
- 质量基线
- 环境说明
- 业务上下文

Reference 不直接决定规则，但能显著提升判断质量。

---

### 7. Routing Layer

对应文件：

- `docs/task-routing-matrix.md`

作用：

- 定义任务如何在 Agent、Standard、Skill、Template 之间进行路由
- 决定默认加载路径和协作顺序

这一层解决的问题是：

> 不是所有任务都应该读取所有文件，必须有一套按需调度机制。

如果没有这一层，仓库就只是“很多文件”，而不是“一个可协作体系”。

---

### 8. Execution Support Layer

对应目录：

- `src/`
- `scripts/`
- `tests/`

作用：

- 提供轻量代码支撑
- 提供最小可验证能力
- 为后续自动化、测试平台、仓库自检等能力留出落点

这一层解决的问题是：

> 如何让这套体系不仅能看，还能逐步进入工程实践。

当前阶段这一层可以很轻，但不能完全没有。

---

## 🔄 Collaboration Logic

整个仓库的协作逻辑可以概括为：

### Step 1：先确定身份

始终以 `AGENTS.md` 规定的默认身份工作。

### Step 2：选择主 Agent

根据任务类型，选择最合适的主 Agent。

### Step 3：加载默认 Standards

为主 Agent 补充其默认需要遵守的 Standards。

### Step 4：按需补充 Skills

当任务进入具体处理层时，再按需加载对应 Skills。

### Step 5：按需使用 Templates

当输出需要固定格式时，再加载 Templates。

### Step 6：按需读取 References

当缺背景信息时，再补充 References。

这套逻辑避免了一开始就“全仓库通读”，从而降低噪音、降低维护成本、提高执行聚焦度。

---

## ✅ Why Not a Single Prompt

本仓库明确不采用“一个超级长 Prompt 解决所有问题”的方式，原因包括：

- 超长 Prompt 难维护
- 多任务场景下容易失焦
- 规则、方法、模板混杂在一起后难复用
- 后续新增能力会导致 Prompt 不断膨胀
- 团队协作和版本管理困难
- 无法形成清晰的结构资产

因此，本仓库采用“身份 + Agent + Standard + Skill + Template + Routing”的分层方式。

---

## 📈 Evolution Strategy

本仓库建议按以下路径持续演进：

### Phase 1：骨架成型

目标：

- 立住身份
- 立住核心 Agent
- 立住核心 Standards
- 立住 Routing 机制

当前你已经基本完成这一阶段。

---

### Phase 2：方法与交付沉淀

目标：

- 补全关键 Skills
- 补全常用 Templates
- 补全架构说明和背景资料
- 建立更明确的任务输入输出结构

---

### Phase 3：工程化落地

目标：

- 增加轻量代码支撑
- 增加自检脚本
- 增加简单测试
- 为接入自动化框架、Agent Workflow、测试平台做准备

---

### Phase 4：平台化演进

目标：

- 接入测试资产管理
- 接入日志与报告分析
- 接入自动化执行体系
- 接入质量门禁与发布检查
- 接入领域知识库和工具链

---

## ⚠️ Design Constraints

这套架构在演进过程中，必须始终遵守以下约束：

- 不破坏“资深测试工程师”这一唯一核心身份
- 不为了看起来完整而堆无效内容
- 不引入职责重复严重的文件
- 不让 Skill 替代 Agent
- 不让 Template 替代 Standard
- 不让 Reference 替代规则和方法
- 不让 Routing 失去清晰性
- 不让仓库退化成无序 Prompt 堆积区

---

## 🏁 Final Requirement

本架构的最终目标不是“做一个好看的仓库”，而是：

> 构建一套真正能够支撑企业级测试任务分析、测试设计、风险判断、质量治理和长期资产沉淀的 AI Agent 协作体系，并为后续工程化和平台化演进打下清晰基础。
