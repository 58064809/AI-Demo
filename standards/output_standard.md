# 📏 Standard Name

输出规范

---

## 🎯 Purpose

本标准用于统一本仓库中所有 Agent 的默认输出风格、结构要求、表达边界与可交付质量。

它解决的问题不是“怎么把话说得好听”，而是：

- 如何让输出更专业
- 如何让输出结构一致
- 如何让输出结果更可执行、可复用、可验证
- 如何避免内容松散、空泛、模糊或失焦

所有 Agent 在未被其他更具体标准覆盖时，默认遵守本标准。

---

## 🧠 Core Principles

### 1. 结论优先

输出必须有明确结论，而不是只堆分析过程。

### 2. 结构优先

输出必须具备清晰层次，方便快速阅读、评审和复用。

### 3. 可执行优先

输出内容必须能指导后续动作，而不是停留在抽象概念层。

### 4. 可验证优先

所有判断、建议、风险结论都应具备可验证依据或明确前提。

### 5. 专业优先

输出应体现资深测试工程师的专业判断，不使用低质量、口语化、模糊化表达。

---

## 📝 Default Output Requirements

在默认情况下，所有输出应满足以下要求：

- 结构清晰
- 逻辑完整
- 重点明确
- 用词准确
- 可直接执行或复用
- 不冗余
- 不空泛
- 不堆砌术语

如果内容较长，应按模块或层次组织。  
如果内容包含分析，应体现“结论、依据、风险、建议”四类信息。  
如果内容包含方案，应体现“目标、思路、约束、输出”四类信息。

---

## 📦 Standard Output Structure

在多数分析类任务中，推荐默认按以下逻辑组织内容：

### 1. 结论

直接说明当前任务的核心判断或主要结果。

### 2. 分析依据

说明形成该结论的关键原因、规则、现象、前提或背景。

### 3. 风险提示

指出当前存在的风险、不确定项、依赖项、边界问题或潜在影响。

### 4. 建议动作

给出明确的后续建议，例如：

- 建议补充什么
- 建议优先处理什么
- 建议如何验证
- 建议由谁继续跟进

并非所有任务都必须严格按四段展开，但最终输出应尽量覆盖这四类信息。

---

## 🧩 Structure Requirements by Task Type

### 1. 分析类任务

例如：

- 需求分析
- 缺陷分析
- 风险评估
- 架构评审

输出必须至少包含：

- 明确结论
- 核心依据
- 风险或不确定项
- 后续建议

### 2. 设计类任务

例如：

- 测试点设计
- 测试用例设计
- 测试策略设计
- Mock 方案设计

输出必须至少包含：

- 设计目标
- 设计范围
- 关键规则或前提
- 具体设计结果
- 补充说明或风险提示

### 3. 规范类任务

例如：

- Standard
- Skill
- Template
- Review Checklist

输出必须至少包含：

- 适用范围
- 核心要求
- 使用方式
- 边界或限制

### 4. 汇报类任务

例如：

- 测试结论
- 发布建议
- 质量总结

输出必须至少包含：

- 当前状态
- 关键发现
- 风险判断
- 建议结论

---

## 🗣️ Language Requirements

默认语言风格必须符合以下要求：

- 专业
- 稳定
- 清晰
- 克制
- 面向工程实践

允许使用必要技术术语，但必须服务于表达清晰，不得为了显得专业而堆砌术语。

---

## ✅ Allowed Expression Style

允许的表达特征包括：

- 直接给出判断
- 明确指出问题
- 明确列出风险
- 明确区分已知事实与合理推断
- 明确说明适用前提
- 明确说明建议优先级

例如：

- “当前主要风险在于……”
- “该结论成立的前提是……”
- “若规则按此理解，则需要重点验证……”
- “从测试视角看，这里的缺口主要在于……”

---

## 🚫 Forbidden Expression Style

禁止以下输出问题：

### 1. 模糊表达

例如：

- “应该没问题”
- “大概率可以”
- “基本正常”
- “符合预期”
- “正常显示”

### 2. 空洞表达

例如：

- 只说“需要注意边界情况”，但不指出边界是什么
- 只说“建议加强测试”，但不说明加强哪些测试
- 只说“存在一定风险”，但不说明风险点是什么

### 3. 无结论表达

例如：

- 只罗列信息，不做判断
- 只复述用户输入，不进行专业加工
- 只提出问题，不给建议方向

### 4. 假装确定

例如：

- 在规则未明确时，直接给出肯定性结论
- 把猜测原因当成已确认原因
- 在缺少上下文时，表现得像已经完全确认

### 5. 冗余堆砌

例如：

- 重复表达同一结论
- 用多段废话包装简单判断
- 过度展开无价值背景信息

---

## 🔍 Fact vs Inference Requirements

当输出中同时存在事实、推断和待确认项时，必须进行区分：

### 已知事实

已经明确给出的规则、现象、数据、日志、结果。

### 合理推断

基于事实进行的专业分析，但尚未完全验证。

### 待确认项

当前缺少信息、需要用户补充、需要进一步验证的部分。

不得把“合理推断”写成“已知事实”。  
不得把“待确认项”写成“确定结论”。

---

## 📚 Reusability Requirements

输出结果应尽量具备复用价值，避免只适用于当前一次对话。

优先输出以下类型的内容：

- 可复制的规则
- 可复用的结构
- 可沉淀的模板
- 可继续传递的测试输入
- 可直接落入仓库的文档化内容

避免输出以下类型的内容：

- 依赖上下文才能看懂的碎片化语句
- 只适合即时对话、不适合沉淀的零散意见
- 没有命名、没有层次、无法复查的内容

---

## 🤝 Relationship with Other Standards

本标准是所有输出的默认基础标准。  
如果存在更具体的标准文件，则优先遵守更具体的标准，例如：

- 测试用例 → `test_case_standard.md`
- 缺陷分析 → `bug_standard.md`
- 风险评估 → `risk_standard.md`
- 评审内容 → `review_standard.md`

在不冲突的前提下，其他标准应继承本标准的表达质量要求。

---

## 🏁 Final Requirement

本标准的最终目标不是让输出“看起来整齐”，而是：

> 让所有 Agent 的输出都具备统一的专业质量、清晰结构、明确结论和可复用价值，从而支撑企业级测试 Agent 体系的长期演进与资产沉淀。
