# ✅ Standard Name

评审规范

---

## 🎯 Purpose

本标准用于统一需求评审、测试评审、方案评审、上线评审等场景下的检查维度、输出方式与评审结论表达，确保评审活动具备一致性、可追溯性和决策价值。

它解决的问题不是“开个会看看”，而是：

- 如何系统发现问题
- 如何统一评审口径
- 如何避免只聊感觉不落结论
- 如何让评审输出真正服务于后续行动和质量决策

所有涉及评审、检查、发布前确认、质量门禁判断的内容，默认遵守本标准。

---

## 🧠 Core Principles

### 1. Issue Discovery First

评审的首要目标是发现问题、缺口和风险，而不是确认“大家都没意见”。

### 2. Structured Review

评审必须有明确维度，不能随意发散或只靠临场经验。

### 3. Decision-Oriented

评审输出必须支持后续决策，例如继续推进、补充信息、补测、阻塞上线等。

### 4. Evidence-Based Judgment

评审结论应尽量基于规则、事实、状态和已有信息，而不是主观印象。

### 5. Actionability First

评审结论必须尽量转化为可执行项，而不是停留在抽象判断。

---

## 📦 Standard Review Structure

默认情况下，评审输出应尽量包含以下信息：

- 评审对象
- 评审范围
- 核心发现
- 风险点
- 待确认项
- 结论
- 后续动作建议

如果是详细评审，可补充：

- 检查项清单
- 问题分级
- 阻塞项与非阻塞项
- Owner 建议
- 时效建议

---

## 📝 Review Dimensions

不同评审场景可关注不同重点，但默认至少应从以下维度检查：

### 1. 完整性

- 需求 / 方案 / 测试内容是否完整
- 是否缺关键规则
- 是否缺关键状态说明
- 是否缺关键输入输出定义

### 2. 一致性

- 文档之间是否一致
- 页面、接口、状态、字段定义是否一致
- 规则之间是否冲突
- 结论与前置假设是否一致

### 3. 可测试性

- 是否具备明确验收标准
- 是否具备可观察结果
- 是否具备足够日志、埋点、状态标识
- 是否能被实际验证

### 4. 风险性

- 是否存在高风险未覆盖项
- 是否存在上线后高概率事故风险
- 是否存在历史高发问题重现风险
- 是否存在关键依赖未确认

### 5. 可执行性

- 评审对象是否能支撑后续开发 / 测试 / 上线动作
- 是否存在无法落地或难以执行的设计
- 是否需要补充澄清、补充数据、补充规则

### 6. 影响范围

- 问题影响哪些角色
- 影响哪些模块
- 是否影响关键链路
- 是否涉及权限、资金、订单、库存、数据一致性等关键对象

## 🔍 Review by Scenario

### 1. Requirement Review

需求评审默认重点关注：

- 目标是否明确
- 规则是否完整
- 状态是否清晰
- 角色与权限是否定义
- 异常与边界是否考虑
- 验收标准是否明确
- 是否存在歧义、遗漏、冲突
- 是否具备可测试性

### 2. Test Design Review

测试设计评审默认重点关注：

- 是否覆盖正常、异常、边界、状态、权限
- 是否存在漏测项
- 是否存在重复设计
- 优先级是否合理
- 用例是否可执行
- 预期结果是否可验证

### 3. Solution / Architecture Review

方案评审默认重点关注：

- 设计是否合理
- 依赖是否明确
- 可维护性与可扩展性是否考虑
- 异常和失败路径是否有兜底
- 是否增加新的质量风险
- 是否便于测试与观察

### 4. Release Review

上线评审默认重点关注：

- 核心测试是否完成
- 高风险问题是否关闭或可接受
- 是否存在阻塞缺陷
- 回归范围是否充分
- 是否有回滚方案
- 是否有监控与告警
- 是否存在未覆盖高风险区域

---

## 📌 Review Conclusion Types

默认建议使用以下类型表达评审结论：

### 1. 可通过

当前信息和质量状态支持继续推进，无明显阻塞项。

### 2. 有条件通过

存在部分问题，但在满足特定前提、补充动作或风险接受条件后可继续推进。

### 3. 不建议通过

当前存在明显缺口、高风险未确认项或阻塞问题，不适合继续推进或上线。

---

## 🚫 Forbidden Writing Style

禁止以下问题：

### 1. 只有讨论，没有结论

例如开完评审后没有明确是否通过、阻塞项是什么、下一步谁处理。

### 2. 只有感觉，没有依据

例如：

- 感觉问题不大
- 看起来可以上线
- 应该没什么风险

### 3. 只提问题，不给后续建议

例如指出规则缺失，但不说明需要补什么、谁补、影响什么。

### 4. 评审范围失焦

例如本来在评审需求，却大量讨论与当前决策无关的扩展设计。

### 5. 不区分阻塞项和非阻塞项

导致后续动作优先级混乱。

---

## ✅ Good Review Characteristics

高质量评审输出通常应具备以下特征：

- 评审对象清楚
- 评审维度清楚
- 问题归类清楚
- 风险等级清楚
- 结论明确
- 后续动作明确
- 可以作为后续跟进依据沉淀下来

---

## 🤝 Relationship with Other Standards

本标准是评审类任务的默认具体标准。

相关场景应结合以下标准共同使用：

- 通用输出要求 → `output_standard.md`
- 测试设计检查 → `test_case_standard.md`
- 缺陷与问题判断 → `bug_standard.md`
- 风险分级与风险判断 → `risk_standard.md`

若评审场景涉及上线门禁，应同时结合测试完成度、缺陷状态和风险接受条件进行判断。

---

## 🏁 Final Requirement

本标准的最终目标不是让评审“显得正式”，而是：

> 让评审活动真正帮助团队系统发现问题、统一判断口径、形成明确结论，并推动后续动作落地，从而提升整体质量决策效率。
