# 🏢 System Context

---

## 🎯 Purpose

本文件用于描述本仓库默认面向的系统背景、测试对象特征、常见业务形态、质量风险类型和协作边界，作为 `references/` 层的基础上下文资料。

它解决的问题包括：

- 这套测试 Agent 仓库默认面对什么样的系统
- 默认假设的测试环境和质量挑战是什么
- 为什么需要需求分析、接口测试、UI 测试、回归测试、性能稳定性分析等多类 Agent
- 为什么本仓库强调风险驱动、状态流转、权限、数据一致性和可测试性

本文件不定义规则，不替代 `AGENTS.md`、`standards/` 或 `skills/`，而是提供统一背景认知。

---

## 🧠 Default System Assumptions

本仓库默认面向的是具有以下一个或多个特征的业务系统：

- 具备真实业务流程的企业系统
- 存在前后端交互的 Web / H5 / App / 小程序系统
- 存在接口调用、数据落库、状态流转和权限控制
- 存在版本迭代、缺陷修复、回归测试和发布活动
- 存在环境、数据、依赖、联调、日志和监控等测试现实问题

它不局限于某一个行业，但默认更适用于：

- 电商
- 企业服务
- 平台类产品
- 中后台系统
- 业务流程型系统
- 多角色、多状态、多规则系统

## 🏗️ Typical System Characteristics

### 1. Multi-Layer Interaction

系统通常具有以下多层交互结构：

- 前端页面 / 客户端
- 接口层
- 服务层
- 数据层
- 外部依赖 / 第三方服务
- 异步任务 / 消息链路（如适用）

这意味着测试不能只盯单点，而要关注从输入到结果的整条链路。

---

### 2. Role and Permission Diversity

系统通常不是单一用户视角，而是包含多个角色，例如：

- 普通用户
- 管理员
- 运营角色
- 审核角色
- 商家 / 服务提供方
- 内部支持角色

这要求测试必须关注：

- 可见范围
- 可操作范围
- 越权 / 漏权 / 错权
- 角色差异下的状态和数据表现

---

### 3. State-Driven Behavior

很多业务系统都不是静态展示，而是依赖状态驱动行为变化，例如：

- 订单状态
- 审批状态
- 任务状态
- 活动状态
- 资源可用状态
- 账号状态

因此，本仓库中的多个 Agent 和 Skill 都强调：

- 状态流转
- 状态边界
- 状态异常
- 非法状态触发
- 状态恢复

---

### 4. Data-Critical Operations

企业系统常常涉及关键数据操作，例如：

- 订单
- 库存
- 价格
- 权益
- 配置
- 报表
- 用户资产
- 敏感字段

这类系统的测试不能只看“界面展示对不对”，还必须关注：

- 数据是否正确写入
- 数据是否正确回滚
- 数据是否一致
- 是否有历史数据影响
- 是否有异步一致性问题

---

### 5. Release and Regression Pressure

默认假设系统存在：

- 持续迭代
- 周期性发版
- 缺陷修复
- 小步快跑式需求更新
- 回归测试压力
- 上线风险判断需求

因此，本仓库特别强调：

- 回归测试代理
- 风险评估规范
- 测试计划 / 测试报告模板
- 发布前质量判断

## ⚠️ Typical Quality Risks

本仓库默认重点关注以下常见风险：

### 1. Requirement Risk

- 需求规则不完整
- 需求存在歧义
- 状态和权限未定义清楚
- 异常和边界缺失
- 验收标准不明确

### 2. Functional Risk

- 主流程可用但异常流程失控
- 状态流转错误
- 页面和接口规则不一致
- 多角色下行为不一致
- 关键流程被小改动破坏

### 3. Data Risk

- 数据重复写入
- 数据漏写
- 缓存和数据库不一致
- 旧数据兼容问题
- 异步结果不一致
- 历史脏数据触发问题

### 4. Permission and Security Risk

- 越权
- 漏权
- 错权
- 敏感功能暴露
- 敏感数据误展示
- 参数篡改绕过限制

### 5. Stability and Dependency Risk

- 外部依赖不稳定
- 接口超时
- 重试放大问题
- 并发冲突
- 限流 / 熔断 / 降级配置不合理
- 日志和监控不足导致问题难定位

### 6. Release Risk

- 缺陷未关闭即上线
- 未覆盖高风险区域
- 回归范围不足
- 发布窗口紧张
- 环境 / 数据 / 依赖未准备就绪

## 🧪 Default Testing Focus

基于上述系统背景，本仓库默认强调以下测试关注点：

- 需求完整性
- 规则明确性
- 状态流转正确性
- 权限边界正确性
- 输入输出正确性
- 异常场景可控性
- 数据一致性
- 回归影响范围
- 发布风险可判断性
- 可测试性与可观测性

这也是为什么本仓库的 Agent 体系会围绕：

- 需求分析
- 测试设计
- API 测试
- UI 测试
- 缺陷分析
- 回归测试
- 性能与稳定性
- 测试架构

来组织。

## 🤝 Collaboration Context

本仓库默认服务于以下协作关系：

- 产品提供需求输入
- 开发负责实现和修复
- 测试负责分析、设计、验证和质量判断
- 运维 / 平台提供环境、部署、监控支持
- 团队共同参与上线判断和风险接受

在这种协作背景下，本仓库特别强调：

- 结构化表达
- 风险驱动分析
- 可执行输出
- 可复用沉淀
- 质量决策支持

## 🧩 Why This Context Matters

本文件存在的意义在于：

- 避免仓库中不同文件假设不同系统背景
- 帮助后续新增 Skill / Template / Agent 时保持统一语义
- 让仓库演进始终围绕真实企业测试问题，而不是偏向抽象 Prompt 仓库
- 为未来接入更真实业务场景提供通用背景基线

## 🚫 Non-Goals

本文件不负责：

- 定义具体业务规则
- 替代某个行业的领域知识文档
- 充当测试标准或评审规范
- 给出具体实现方案

如果未来接入某个明确行业（如电商、金融、企业服务）的专门知识，应在 `references/` 中额外补充领域上下文文件。
