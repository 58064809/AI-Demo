# 📚 Glossary

---

## 🎯 Purpose

本文件用于统一本仓库中常见测试术语、架构术语、质量术语和协作术语的含义，减少不同 Agent、不同文档、不同模板之间的表达偏差和理解歧义。

它解决的问题包括：

- 同一个词在不同文件中是否表达一致
- 不同 Agent 使用同一术语时是否语义统一
- 仓库中的层次概念、任务概念和测试概念是否可复查
- 后续新增内容时是否有统一术语参考

本文件属于 `references/`，用于补充上下文，不替代 `AGENTS.md`、`standards/`、`skills/` 或 `templates/` 的规则性内容。

---

## 🧠 Repository Terms

### Agent

在本仓库中，Agent 指面向某类测试任务场景的专业角色分身。  
它定义的是：

- 由谁处理该类任务
- 如何思考该类任务
- 默认输出什么
- 与其它 Agent 如何协作

本仓库中的多个 Agent，不代表多个不同职业，而是同一名**资深测试工程师**在不同任务场景下的专业分身。

---

### Standard

Standard 指共享规范。  
它定义的是：

- 输出应该满足什么要求
- 质量判断应遵守什么口径
- 用例、缺陷、风险、评审等内容的统一标准是什么

Standard 解决的是“按什么标准做”的问题。

---

### Skill

Skill 指能力方法手册。  
它定义的是：

- 某类任务具体应该怎么做
- 处理步骤是什么
- 默认检查维度有哪些
- 常见问题和易错点有哪些

Skill 解决的是“具体怎么做”的问题。

---

### Template

Template 指固定交付模板。  
它定义的是：

- 某类输出通常长什么样
- 应该按什么结构交付
- 哪些字段或栏目是默认应包含的

Template 是交付载体，不替代 Standard。

---

### Reference

Reference 指参考资料。  
它用于沉淀：

- 长期背景信息
- 术语解释
- 系统上下文
- 环境信息
- 质量基线

Reference 提供背景支持，但不直接充当规则。

---

### Routing

Routing 指任务路由。  
它描述的是：

- 某类任务优先应该选择哪个 Agent
- 默认应该加载哪些 Standard
- 哪些 Skill / Template 是按需补充的
- 是否需要引入次级 Agent

Routing 解决的是“任务如何在仓库内协作流转”的问题。

---

### Architecture Overview

Architecture Overview 指仓库整体架构说明。  
它描述仓库的：

- 分层设计
- 各层职责
- 协作方式
- 演进思路

它不负责详细规则，而负责整体设计说明。

---

### Repository Roadmap

Repository Roadmap 指仓库建设路线图。  
它描述的是：

- 当前完成度
- 阶段划分
- 优先级顺序
- 后续扩展方向

它解决“仓库当前做到哪、后面怎么做”的问题。

---

## 🧪 Testing Terms

### Requirement Review

需求评审，指在开发和测试执行前，对需求目标、规则、状态、权限、边界、异常和可测试性进行结构化检查的活动。  
目标是更早发现遗漏、歧义、冲突和高风险点。

---

### Test Design

测试设计，指把需求、规则、流程和风险转化为测试点和测试用例的过程。  
重点不只是覆盖主流程，还包括异常、边界、状态流转、权限和数据一致性。

---

### Test Case

测试用例，指围绕单一验证目标设计的可执行测试条目。  
通常应包含：

- 标题
- 优先级
- 前置条件
- 测试步骤
- 预期结果

---

### Test Plan

测试计划，指测试活动开始前的整体规划。  
通常包括：

- 测试目标
- 测试范围
- 测试策略
- 环境与数据
- 风险与依赖
- 进度安排
- 准入 / 准出标准

---

### Test Report

测试报告，指测试活动结束后对执行情况、问题、风险和结论的结构化总结。  
它服务于：

- 质量判断
- 风险判断
- 发布决策
- 后续动作安排

---

### Smoke Test

冒烟测试，指在构建后或发布前快速验证核心链路是否可用的最小测试集合。  
它通常强调：

- 核心流程
- 快速反馈
- 高信噪比
- 最小闭环

---

### Regression Test

回归测试，指在需求变更、代码改动、缺陷修复或版本发布前后，对已有功能进行的影响验证。  
重点是：

- 识别改动影响范围
- 优先覆盖高风险区域
- 控制测试成本
- 支撑发布判断

---

### API Testing

接口测试，指围绕接口契约、业务规则、状态流转、鉴权、幂等、异常处理和数据一致性进行的验证活动。  
接口测试不只看是否返回 200，还要看结果是否正确。

---

### UI Testing

UI 测试，指围绕页面展示、交互、反馈、流程跳转、角色差异和状态变化进行的验证活动。  
UI 测试不只验证“能打开”，还要验证“能正确交互、正确反馈、正确联动”。

---

### Performance Testing

性能测试，指围绕系统在不同负载条件下的响应时间、吞吐能力和资源使用情况进行的验证活动。  
重点包括：

- 并发能力
- 响应时间
- 资源瓶颈
- 容量边界

---

### Stability Testing

稳定性测试，指围绕系统在长时间运行、异常依赖、流量波动和资源紧张条件下的可持续运行能力进行的验证活动。  
重点包括：

- 长稳运行
- 容错能力
- 恢复能力
- 异常场景下的韧性

---

### Bug

Bug 指系统实际行为与预期行为之间存在差异，且该差异会带来功能、数据、状态、权限、性能、稳定性或体验层面的异常结果。

---

### Bug Triage

Bug Triage 指缺陷分级、归因、影响评估和上线影响判断过程。  
它不是简单“打个标签”，而是把问题转化为可决策、可流转、可回归的质量项。

---

### Root Cause

根因，指问题真正发生的根本原因。  
需要注意区分：

- 表面现象
- 触发条件
- 传播路径
- 可能根因
- 已确认根因

---

### Workaround

绕过方案，指在问题未修复前，用户或团队临时采用的替代操作方式。  
判断缺陷等级时，不能只看“是否能绕过”，还要看绕过成本和副作用。

---

## ⚖️ Quality Terms

### Risk

风险，指某个问题、缺口或不确定项在未来触发并造成业务影响、用户影响、数据影响或发布影响的可能性。

风险通常需要同时考虑：

- 发生概率
- 影响程度

---

### Severity

严重程度，指问题本身的影响级别。  
通常用于 Bug 分级，例如：

- P0
- P1
- P2
- P3

它描述的是“问题有多严重”。

---

### Priority

优先级，指问题或任务应被处理的先后顺序。  
它描述的是“当前应该先做哪个”。

严重程度和优先级相关，但不完全相同。

---

### Blocking Issue

阻塞项，指当前问题严重到会阻断测试推进、需求推进、发布推进或上线决策的项目。  
阻塞项不一定只由严重 Bug 构成，也可能来自规则缺失、依赖不可用或高风险未确认项。

---

### Observability

可观测性，指系统是否具备足够的日志、埋点、状态标识、监控和错误信息，使问题在发生后能够被及时发现、定位和分析。

---

### Testability

可测试性，指需求、设计或系统实现是否便于测试验证。  
例如：

- 是否有明确输入输出
- 是否有明确验收标准
- 是否有可观察结果
- 是否便于构造测试场景

---

### Data Consistency

数据一致性，指页面展示、接口返回、缓存状态、数据库状态和上下游系统状态之间是否保持一致。

---

### Idempotency

幂等性，指在同一操作被重复触发时，系统最终结果保持一致，不产生额外副作用。

---

### Contract

契约，通常指接口契约或系统交互契约。  
包括：

- 输入定义
- 输出定义
- 字段约束
- 错误码
- 状态码
- 行为规则

---

### Mock

Mock 指用于替代真实依赖系统的模拟服务或模拟返回。  
它的目标不是“伪造成功结果”，而是为联调、测试、自动化和异常模拟提供可控、可重复的替身能力。

---

## 🧰 Engineering Terms

### Execution Support Layer

执行支撑层，通常指：

- `src/`
- `scripts/`
- `tests/`

这一层的作用是让仓库具备最小工程化能力，而不仅停留在文档层。

---

### Smoke Check

Smoke Check 指最小自检脚本，用于快速确认仓库核心能力是否处于可用状态。  
在当前仓库中，它主要用来验证：

- 架构快照是否可读取
- 核心统计是否正常
- 基础工程入口是否可用

---

### Architecture Snapshot

架构快照，指对仓库当前结构状态的结构化描述。  
在当前仓库中，`context.py` 通过动态扫描方式生成当前架构快照，用于：

- 脚本自检
- 测试校验
- 后续工程化扩展

---

## 🚫 Term Usage Notes

在本仓库中，以下做法应尽量避免：

- 把 Agent 和 Skill 混用
- 把 Template 当成 Standard
- 把 Reference 当成规则来源
- 把 Severity 和 Priority 混为一谈
- 把 Smoke Test 和 Regression Test 混为一谈
- 把“返回成功”直接等同于“测试通过”
- 把“存在绕过方案”直接等同于“问题不严重”

---

## 🏁 Final Requirement

本文件的最终目标不是“收集一堆名词解释”，而是：

> 让本仓库中的核心术语具备统一口径、统一语义和统一边界，从而减少沟通歧义，提升 Agent 协作一致性和仓库长期可维护性。
