# 🛠️ Skill Name

测试用例设计技能

---

## 🎯 Purpose

本技能用于定义测试点设计和测试用例设计的具体处理方法，帮助 Agent 在面对需求、规则、页面、接口、状态流转和复杂业务逻辑时，能够以结构化、系统化、可复用的方式完成测试设计。

它解决的问题不是“列几个场景”，而是：

- 如何把需求转成测试点
- 如何系统补齐正常、异常、边界、状态、权限、数据一致性场景
- 如何控制覆盖深度与测试成本
- 如何避免重复设计和无效堆量
- 如何输出真正可执行、可复查、可维护的测试内容

---

## 🧠 Core Principles

### 1. Rule Before Case

先理解规则，再设计用例。  
测试用例不是把需求原文改写一遍，而是把规则转成可验证场景。

### 2. Coverage with Focus

覆盖要尽量完整，但不能平均用力。  
应优先覆盖高风险、高影响、高频路径和关键异常场景。

### 3. Scenario-Oriented Thinking

测试设计应围绕“场景”而不是“字段列表”展开，重点关注用户行为、状态变化、输入输出和系统反馈。

### 4. Verifiable Output

测试用例最终必须可执行、可验证、可复查，而不是抽象提醒或口号式检查项。

### 5. Reuse and Maintainability

测试内容应具备复用价值，避免大量低差异重复用例，避免后期维护失控。

---

## 📦 Applicable Scenarios

本技能适用于但不限于以下场景：

- 根据需求设计测试点
- 根据规则设计测试用例
- 根据 PRD / 接口文档 / 页面设计稿补测试覆盖
- 针对缺陷做补测设计
- 针对改动做回归集设计
- 对已有用例进行补漏和去重
- 输出适合手工执行或自动化改造的测试内容

---

## 🔍 Input Requirements

开始设计测试用例前，尽量先明确以下输入：

- 功能目标是什么
- 规则是否完整
- 有哪些角色
- 有哪些状态
- 输入和输出是什么
- 页面、接口、数据库或其它观察点有哪些
- 是否存在关键异常路径
- 是否存在高风险业务对象（资金、库存、订单、权限、敏感数据等）
- 当前任务目标是“完整设计”还是“快速回归”

如果上述信息不完整，应先显式列出前提和待确认项，再继续设计。

## 🧩 Standard Design Process

### Step 1：Clarify the Test Object

先明确当前设计对象是什么：

- 一个功能
- 一个页面
- 一个接口
- 一个状态流转
- 一条业务链路
- 一次缺陷修复
- 一次版本改动

不要在对象不清时直接开始写测试用例。

---

### Step 2：Extract Rules

梳理规则，包括但不限于：

- 输入规则
- 状态规则
- 权限规则
- 依赖规则
- 流程规则
- 数量 / 金额 / 时间 / 次数限制
- 成功条件
- 失败条件
- 异常处理规则

如果规则不清，必须先标记。

---

### Step 3：Split into Test Dimensions

默认从以下维度拆分：

- 正常流程
- 异常流程
- 边界条件
- 非法输入
- 状态流转
- 权限控制
- 数据一致性
- 重复操作 / 并发 / 幂等（如适用）

这一步的目标不是直接写表格，而是先确认测试覆盖面。

---

### Step 4：Determine Priority

根据以下因素判断优先级：

- 是否核心业务链路
- 是否高频使用
- 是否高风险对象
- 是否容易引发线上事故
- 是否历史高发问题区域
- 是否本次改动重点区域

优先保证 P0 / P1 场景先完整。

---

### Step 5：Organize into Test Cases

将测试点转为测试用例时，应确保：

- 一条用例只聚焦一个核心验证目标
- 步骤可执行
- 预期可验证
- 用例边界清晰
- 同类场景不过度重复

---

### Step 6：Review for Gaps and Redundancy

完成初稿后，至少检查两件事：

#### 是否有漏测

重点检查：

- 异常是否漏了
- 边界是否漏了
- 状态切换是否漏了
- 权限差异是否漏了
- 数据变化是否漏了

#### 是否有重复

重点检查：

- 同一规则是否被重复表达
- 多条用例是否只是在换说法
- 低价值场景是否过多占比

---

## 📌 Default Coverage Checklist

设计测试点时，默认优先检查以下覆盖项：

### 1. Normal Flow

- 主流程是否成功
- 合法输入是否通过
- 成功后状态是否正确变化
- 成功后数据是否正确更新

### 2. Exception Flow

- 前置条件不满足如何处理
- 外部依赖异常如何处理
- 操作失败是否有明确反馈
- 中断、重试、刷新、返回后是否异常

### 3. Boundary Conditions

- 最小值
- 最大值
- 临界值
- 空值
- Null
- 特殊字符
- 超长值
- 次数 / 时间 / 金额 / 数量边界

### 4. Invalid Input

- 缺字段
- 错类型
- 错格式
- 非法组合
- 越权输入
- 绕过限制输入

### 5. State Transition

- 状态进入是否正确
- 状态流转是否正确
- 是否允许重复触发
- 是否允许逆向流转
- 中断后是否可恢复

### 6. Permission Control

- 不同角色可见范围是否正确
- 不同角色可操作范围是否正确
- 是否存在越权、漏权、错权问题

### 7. Data Consistency

- 页面 / 接口 / 数据库是否一致
- 是否存在重复写入、漏写、脏数据
- 多步骤后数据累计、回滚、同步是否正确

### 8. Repeat / Concurrency / Idempotency

- 重复点击
- 重复提交
- 接口重试
- 并发请求
- 资源竞争
- 幂等控制

## 🧪 Design Strategies by Scenario

### 1. For Form/Input Scenarios

重点关注：

- 输入格式
- 必填校验
- 长度 / 范围
- 非法字符
- 联动字段约束
- 提交前与提交后状态变化

### 2. For Workflow Scenarios

重点关注：

- 主流程
- 分支流程
- 回退路径
- 中断恢复
- 状态切换
- 角色流转
- 数据落点

### 3. For State Machine Scenarios

重点关注：

- 合法状态流转
- 非法状态流转
- 重复触发
- 逆向触发
- 中间态和异常态

### 4. For Defect Fix Verification

重点关注：

- 原问题是否修复
- 同类问题是否被顺手引入
- 修复点上下游是否受影响
- 历史相关问题是否需要补回归

### 5. For Rapid Regression Design

重点关注：

- 改动直接影响点
- 高风险上下游
- 核心链路兜底
- 关键历史问题区域

## 📝 Output Guidance

最终输出时，应遵守：

- `standards/test_case_standard.md`
- `templates/test_case_template.md`

如输出测试点，可按模块或规则分类。  
如输出测试用例，应保证列结构统一、步骤清晰、预期明确。

---

## 🚫 Common Mistakes

设计测试用例时，必须避免以下问题：

- 只写主流程，不写异常和边界
- 只看输入，不看状态和数据结果
- 只列字段，不列场景
- 用例目标不聚焦，一条用例验证多个核心目标
- 步骤不可执行
- 预期不可验证
- 同一规则被反复写成很多低差异用例
- 规则不清却假装已经明确

---

## ✅ Good Output Characteristics

高质量测试设计通常具备以下特征：

- 测试对象清晰
- 规则拆解清晰
- 覆盖维度完整
- 优先级合理
- 用例结构统一
- 可直接执行
- 可直接用于评审
- 可直接作为自动化改造输入

## 🏁 Final Requirement

本技能的最终目标不是“多写一些测试用例”，而是：

> 让测试设计过程具备清晰方法、完整覆盖、合理取舍和长期复用价值，使测试用例真正服务于质量保障，而不是沦为形式化产物。
