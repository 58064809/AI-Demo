# 🛠️ Skill Name

自动化脚本设计技能

---

## 🎯 Purpose

本技能用于定义测试自动化脚本设计、自动化用例落地、可维护性控制、执行稳定性保障与自动化资产沉淀的具体处理方法，帮助 Agent 在面对接口自动化、UI 自动化、回归自动化和辅助测试脚本设计时，能够以结构化、系统化、可复用的方式完成高质量自动化脚本方案设计。

它解决的问题不是“把手工步骤翻译成代码”，而是：

- 如何判断哪些内容值得自动化
- 如何让自动化脚本具备可维护性、稳定性和复用性
- 如何避免自动化脚本脆弱、耦合严重、跑不稳
- 如何让自动化脚本真正服务于回归效率和质量保障
- 如何把自动化能力沉淀为长期测试资产

---

## 🧠 Core Principles

### 1. Automation Follows Test Value

不是所有测试都值得自动化。  
应优先自动化那些高频、稳定、可重复、回归价值高、人工成本高的场景。

### 2. Stability Before Scale

自动化脚本首先要稳定，其次才是扩大覆盖范围。  
不稳定的自动化只是噪音放大器。

### 3. Reuse over Copying

优先设计公共能力、可复用组件、通用断言和统一数据管理，而不是大量复制粘贴脚本。

### 4. Maintainability as a First-Class Goal

脚本设计必须考虑长期维护成本，包括：

- 变更适应能力
- 定位问题能力
- 运行失败可读性
- 公共层复用能力

### 5. Observability and Diagnosis

自动化脚本不是只要“能跑”，还必须在失败时：

- 容易看懂
- 容易定位
- 容易复现
- 容易区分是脚本问题还是产品问题

## 📦 Applicable Scenarios

本技能适用于但不限于以下场景：

- 接口自动化脚本设计
- UI 自动化脚本设计
- 回归自动化方案设计
- 冒烟自动化设计
- 数据初始化与前置步骤自动化
- 自动化测试框架分层设计
- 脚本可维护性优化
- 自动化失败诊断能力设计

---

## 🔍 Input Requirements

开始设计自动化脚本前，尽量先明确以下输入：

- 当前要自动化的目标是什么
- 当前场景是否高频且稳定
- 当前规则是否足够明确
- 是否已有可复用测试用例
- 是否有稳定测试环境
- 是否有可控测试数据
- 是否依赖第三方系统或不稳定接口
- 是否适合串行 / 并行执行
- 失败后需要哪些日志、截图、响应信息用于诊断
- 当前自动化是为了冒烟、回归、持续集成还是专项验证

如果这些输入不明确，应先补前提和约束。

## 🧩 Standard Design Process

### Step 1：Decide Whether to Automate

先判断当前场景是否值得自动化，重点看：

- 使用频率是否高
- 回归价值是否高
- 规则是否稳定
- 执行步骤是否重复
- 人工执行成本是否高
- 环境 / 数据是否可控

如果规则不稳定、环境不稳定、数据不可控，强行自动化往往收益低。

---

### Step 2：Clarify Automation Scope

明确自动化范围，例如：

- 单接口验证
- 关键业务链路验证
- 页面核心流程验证
- 构建后冒烟
- 发布前回归
- 缺陷修复补回归
- 公共能力脚本

不要在范围不清时直接开始设计脚本。

---

### Step 3：Split Script Layers

根据场景，建议拆分以下层次：

- 配置层
- 数据层
- 请求 / 页面操作层
- 断言层
- 用例编排层
- 报告 / 日志层

目标是避免把所有逻辑堆进单个测试脚本。

---

### Step 4：Define Reusable Components

识别哪些内容适合沉淀为公共能力，例如：

- 通用请求方法
- 通用页面操作
- 通用等待逻辑
- 通用断言方法
- 通用数据构建方法
- 通用环境配置
- 通用日志与报告能力

---

### Step 5：Design Test Data and Dependency Handling

明确：

- 脚本依赖哪些数据
- 数据如何初始化
- 是否需要 Mock
- 是否需要独立账号 / 独立资源
- 是否需要执行前清理或执行后回收
- 如何避免并发执行相互污染

---

### Step 6：Design Failure Diagnosis Capability

每个自动化脚本至少要考虑失败后如何定位，例如：

- 接口自动化：打印请求 / 响应 / 断言差异
- UI 自动化：截图 / 页面状态 / 控件定位信息
- 数据验证：打印关键数据快照
- 流程自动化：打印关键步骤执行状态

---

## 📌 Default Design Checklist

设计自动化脚本时，默认优先检查以下内容：

### 1. Suitability

- 是否值得自动化
- 是否属于高频回归场景
- 是否具备足够稳定性
- 是否具备可重复执行条件

### 2. Scope Clarity

- 自动化范围是否清晰
- 是否聚焦核心目标
- 是否避免过度自动化

### 3. Layering and Decoupling

- 是否分层清晰
- 是否避免脚本中硬编码大量逻辑
- 是否避免测试数据、请求逻辑、断言逻辑混在一起

### 4. Reusability

- 是否存在可抽象公共组件
- 是否减少重复代码
- 是否便于新增用例复用

### 5. Data Management

- 是否具备稳定数据准备方式
- 是否有清理 / 重置策略
- 是否适合并发执行
- 是否容易出现相互污染

### 6. Assertion Quality

- 断言是否真正覆盖关键结果
- 是否只断言表面现象
- 是否有状态、数据和业务结果断言
- 是否便于失败后理解原因

### 7. Diagnosability

- 失败时能否快速知道失败在哪
- 是否有足够日志
- 是否有必要截图、响应体、步骤标记
- 是否易于区分脚本问题和产品问题

### 8. Maintenance Cost

- 规则变更后是否容易调整
- 页面小改版后是否容易适配
- 公共层变更是否会影响过大
- 是否有明显脆弱点

## 🧪 Design Strategies by Scenario

### 1. For API Automation

重点关注：

- 公共请求封装
- 环境和鉴权管理
- 数据准备与清理
- 响应断言
- 状态 / 数据一致性断言
- 幂等与异常场景覆盖

### 2. For UI Automation

重点关注：

- 页面元素稳定定位
- 等待机制
- 页面状态检查
- 关键业务流程覆盖
- 失败截图与操作日志
- 减少对脆弱 UI 细节的过度依赖

### 3. For Smoke Automation

重点关注：

- 核心链路最小闭环
- 运行速度
- 稳定性
- 快速反馈能力
- 失败后高信噪比输出

### 4. For Regression Automation

重点关注：

- 高频回归场景
- 公共组件复用
- 数据隔离
- 并行执行安全性
- 覆盖与维护成本平衡

### 5. For Utility / Support Scripts

重点关注：

- 数据造数脚本
- 数据清理脚本
- 环境检查脚本
- 日志辅助分析脚本
- 报告汇总脚本

## 🚫 Common Mistakes

设计自动化脚本时，必须避免以下问题：

- 不判断价值，见场景就自动化
- 只把手工步骤翻译成代码，不做抽象
- 所有逻辑写进一个脚本文件
- 没有数据管理策略
- 没有失败诊断能力
- 断言只看是否返回 200 / 页面是否打开
- 环境和依赖不稳定却强行扩大自动化覆盖
- 自动化脚本越来越多，但没人能维护
