# 🛠️ Skill Name

需求评审技能

---

## 🎯 Purpose

本技能用于定义需求评审与需求分析的具体处理方法，帮助 Agent 在面对 PRD、流程图、原型图、接口规则、页面说明和业务规则时，能够以结构化、系统化、可复用的方式完成高质量需求评审。

它解决的问题不是“把需求看一遍”，而是：

- 如何发现需求中的遗漏、歧义、冲突和不合理设计
- 如何系统梳理功能规则、状态规则、权限规则和边界条件
- 如何识别需求的可测试性问题
- 如何把需求评审结果沉淀为后续测试设计、开发实现和质量决策的高质量输入
- 如何避免评审流于表面、只聊感觉、不落结论

---

## 🧠 Core Principles

### 1. Goal Before Detail

先理解需求想解决什么问题，再进入细节，不要一开始就盯字段和页面细节而忽略业务目标。

### 2. Rule Before Opinion

需求评审必须围绕规则、流程、状态、权限、边界和可验证性展开，而不是停留在主观喜好层面。

### 3. Risk-Oriented Review

优先识别高风险、高影响、高复杂度、高不确定性的需求点，而不是平均分配注意力。

### 4. Testability Awareness

需求不仅要“能做”，还要“能测、能验、能观察、能回归”。

### 5. Structured Conclusion

评审必须输出明确问题、明确风险、明确待确认项和明确后续建议，不能只形成零散讨论记录。

## 📦 Applicable Scenarios

本技能适用于但不限于以下场景：

- PRD 评审
- 需求澄清
- 原型评审
- 测试前置分析
- 规则梳理
- 研发联调前的需求确认
- 上线前的规则核查
- 复杂功能改版的影响分析

## 🔍 Input Requirements

开始需求评审前，尽量先明确以下输入：

- 需求目标是什么
- 业务背景是什么
- 涉及哪些角色
- 涉及哪些页面、接口、流程或状态
- 成功标准和失败标准是什么
- 是否有原型图、流程图、字段说明、状态说明
- 是否有历史版本行为或兼容性要求
- 是否有上线时间、范围、优先级等背景约束

如果输入不完整，应先明确前提和待确认项，再进入评审。

## 🧩 Standard Review Process

### Step 1：Clarify Business Goal

先明确：

- 这个需求要解决什么问题
- 为什么现在做
- 成功后对业务、用户、流程会带来什么变化

如果目标不清，后续规则很容易看偏。

---

### Step 2：Split Functional Scope

拆解功能范围，包括：

- 核心功能点
- 子功能点
- 角色差异
- 页面入口
- 接口入口
- 主流程
- 分支流程
- 失败路径

这一步的目标是先把“做什么”拆出来。

---

### Step 3：Extract Explicit and Implicit Rules

梳理规则，包括：

- 显性规则
- 隐性规则
- 前置条件
- 限制条件
- 角色权限规则
- 状态流转规则
- 数据处理规则
- 时间 / 次数 / 金额 / 数量限制
- 成功和失败判定规则

需求评审不能只看页面文案，要主动找隐含规则。

---

### Step 4：Check Key Review Dimensions

默认从以下维度检查：

- 目标是否明确
- 范围是否完整
- 规则是否闭环
- 流程是否清晰
- 状态是否完整
- 权限是否明确
- 边界是否覆盖
- 异常是否定义
- 可测试性是否具备
- 验收标准是否可执行

---

### Step 5：Identify Risks and Gaps

识别：

- 规则缺失
- 逻辑冲突
- 边界遗漏
- 状态遗漏
- 角色遗漏
- 异常遗漏
- 可测试性问题
- 可观察性不足
- 上下游依赖风险
- 历史兼容性风险

---

### Step 6：Form Structured Outputs

最终至少形成以下输出：

- 功能点拆解
- 关键规则梳理
- 测试关注点
- 风险点
- 需求问题清单
- 待确认项
- 建议补充项

---

## 📌 Default Review Checklist

评审需求时，默认优先检查以下内容：

### 1. Goal and Scope

- 需求目标是否清晰
- 范围边界是否明确
- 是否说明本次做什么、不做什么

### 2. Functional Logic

- 主流程是否清楚
- 分支流程是否清楚
- 流程之间是否衔接合理
- 是否存在逻辑断层

### 3. Input and Output

- 输入项是否定义完整
- 输出结果是否可观察
- 成功和失败结果是否明确
- 字段规则是否完整

### 4. State Transition

- 是否有状态
- 状态是否定义清楚
- 状态切换条件是否明确
- 非法状态是否有约束
- 重复触发和逆向流转是否定义

### 5. Role and Permission

- 涉及哪些角色
- 每个角色能看什么
- 每个角色能做什么
- 是否存在权限边界不清问题

### 6. Exception and Boundary

- 空值、上限、下限、特殊值是否定义
- 失败路径是否定义
- 中断、重试、重复操作是否定义
- 外部依赖异常时是否定义处理方式

### 7. Data and Consistency

- 涉及哪些关键数据
- 数据由谁创建 / 修改 / 消费
- 是否存在同步 / 异步 / 缓存一致性要求
- 是否有修数、补偿、回滚需求

### 8. Testability and Observability

- 是否具备明确验收标准
- 是否具备可观察结果
- 是否具备日志、埋点、状态标识
- 是否能设计出可执行测试用例

## 🧪 Review Strategies by Scenario

### 1. For New Feature Review

重点关注：

- 功能范围是否完整
- 规则是否闭环
- 异常是否考虑
- 是否遗漏角色和状态

### 2. For Feature Change Review

重点关注：

- 新旧行为差异
- 兼容性影响
- 上下游联动影响
- 历史逻辑是否被破坏

### 3. For Workflow Review

重点关注：

- 主流程
- 分支流程
- 回退路径
- 状态切换
- 中断恢复
- 权限流转

### 4. For UI + API Mixed Review

重点关注：

- 页面规则与接口规则是否一致
- 页面提示与接口结果是否一致
- 状态定义是否统一
- 数据展示与后台处理是否一致

### 5. For Release-Critical Review

重点关注：

- 核心链路是否存在高风险未确认项
- 是否存在阻塞上线的规则缺失
- 是否存在高影响未定义异常场景

## 🚫 Common Mistakes

需求评审时，必须避免以下问题：

- 只复述需求，不做分析
- 只看主流程，不看异常和边界
- 只看页面，不看规则和状态
- 规则缺失却默认开发会补齐
- 不区分已知规则和待确认项
- 只提问题，不形成结论和建议
- 只从实现角度看，不从测试和质量角度看
- 不关注可测试性和可观察性

---

## ✅ Good Output Characteristics

高质量需求评审通常具备以下特征：

- 功能拆解清晰
- 规则梳理完整
- 风险点明确
- 问题清单可执行
- 待确认项清晰
- 能直接作为测试设计输入
- 能帮助开发减少误解
- 能帮助上线前降低高风险遗漏

## 🏁 Final Requirement

本技能的最终目标不是“把需求看完”，而是：

> 让需求评审具备清晰方法、统一维度、明确结论和长期复用价值，帮助团队更早发现问题、更准确理解规则，并为后续测试设计和质量保障提供高质量输入。
