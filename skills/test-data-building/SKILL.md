# 🛠️ Skill Name

测试数据构建技能

---

## 🎯 Purpose

本技能用于定义测试数据设计、测试数据构建、测试数据隔离、测试数据清理与测试数据复用的具体处理方法，帮助 Agent 在面对功能测试、接口测试、回归测试、自动化测试、联调测试和异常场景验证时，能够以结构化、系统化、可复用的方式完成高质量测试数据准备。

它解决的问题不是“随便造几条数据”，而是：

- 如何让测试数据真正支撑测试目标
- 如何构建可控、可重复、可隔离的测试数据
- 如何为正常、异常、边界、权限、状态流转等场景准备对应数据
- 如何减少环境脏数据、历史数据、随机数据对测试结论的干扰
- 如何让测试数据成为长期可复用的测试资产

---

## 🧠 Core Principles

### 1. Data Supports Test Goal

测试数据不是独立存在的，它必须服务于明确的测试目标。  
先有测试场景，再有测试数据，而不是先造数据再找场景。

### 2. Controllability First

测试数据应尽量可控、可预测、可重复使用。  
不可控的数据会直接降低测试稳定性和结果可信度。

### 3. Isolation over Pollution

优先设计可隔离的数据，避免不同测试任务之间互相污染、互相抢占状态。

### 4. Full Lifecycle Awareness

测试数据不仅要能创建，还要考虑：

- 如何初始化
- 如何流转
- 如何回收
- 如何重置
- 如何复用

### 5. Scenario Coverage

测试数据必须覆盖：

- 正常数据
- 异常数据
- 边界数据
- 状态型数据
- 权限型数据
- 脏数据 / 历史数据 / 冲突数据（如适用）

---

## 📦 Applicable Scenarios

本技能适用于但不限于以下场景：

- 功能测试前的数据准备
- 接口测试中的请求前置数据准备
- 自动化测试中的数据初始化与回收
- 回归测试中的高频基础数据准备
- 权限测试中的角色与账号准备
- 状态流转测试中的阶段性数据准备
- Mock 联调中的假数据生成
- 缺陷复现与修复验证中的特殊数据构造

---

## 🔍 Input Requirements

开始构建测试数据前，尽量先明确以下输入：

- 测试目标是什么
- 需要验证哪些规则
- 涉及哪些角色
- 涉及哪些状态
- 需要哪些前置业务对象
- 数据由哪些系统创建 / 修改 / 消费
- 是否依赖外部系统或异步任务
- 是否存在唯一性约束
- 是否存在时间、次数、金额、数量边界
- 测试结束后数据是否需要回收或重置

如果这些信息不完整，应先明确前提和待确认项。

## 🧩 Standard Design Process

### Step 1：Clarify Test Objective

先明确当前测试数据是为了支持什么场景：

- 正常流程验证
- 异常流程验证
- 边界场景验证
- 状态流转验证
- 权限验证
- 并发 / 幂等验证
- 回归验证
- 自动化执行

不同目标会决定数据复杂度和准备方式。

---

### Step 2：Identify Required Business Objects

梳理本次测试需要哪些业务对象，例如：

- 用户
- 角色
- 商品
- 订单
- 优惠券
- 审批单
- 配置项
- 任务记录
- 报表数据

目标是先明确“要准备什么对象”，而不是直接开始写 SQL 或调用接口。

---

### Step 3：Determine Required States

为每个业务对象明确需要哪些状态，例如：

- 新建态
- 可用态
- 已使用态
- 已禁用态
- 已过期态
- 待支付 / 已支付 / 已取消
- 待审核 / 审核中 / 已驳回 / 已通过

如果不先定义状态，后面很容易出现“有数据但状态不对”。

---

### Step 4：Split Data Types by Scenario

默认从以下维度拆分数据类型：

- 正常数据
- 异常数据
- 边界数据
- 空值 / Null 数据
- 非法格式数据
- 权限差异数据
- 状态差异数据
- 冲突数据
- 历史遗留数据
- 并发竞争数据（如适用）

---

### Step 5：Choose Data Construction Method

根据场景选择合适方式，例如：

- 前台手工创建
- 后台配置创建
- 接口造数
- SQL 造数
- 脚本造数
- Mock 返回
- 固定初始化数据
- 自动化前置步骤造数

原则是优先选择：

- 稳定
- 可重复
- 易回收
- 易理解

的方法。

---

### Step 6：Define Reuse and Cleanup Strategy

必须明确：

- 这批数据是否可复用
- 是否需要唯一隔离
- 是否需要执行后清理
- 是否需要测试前重置
- 是否需要数据回收脚本
- 是否存在脏数据残留风险

---

## 📌 Default Data Design Checklist

构建测试数据时，默认优先检查以下内容：

### 1. Object Completeness

- 关键业务对象是否齐全
- 是否缺少前置依赖对象
- 对象之间关联关系是否正确

### 2. State Accuracy

- 数据状态是否与测试目标匹配
- 是否具备需要的前置状态
- 是否能支撑后续状态流转

### 3. Role and Permission Fit

- 不同角色是否有对应账号或对象
- 是否能验证权限边界
- 是否能验证越权 / 漏权场景

### 4. Boundary Coverage

- 是否有最小值 / 最大值 / 临界值数据
- 是否有空值 / Null / 特殊值数据
- 是否有非法格式 / 非法组合数据

### 5. Uniqueness and Conflict

- 是否存在唯一键冲突
- 是否存在重复数据冲突
- 是否能构造同名 / 同号 / 同时间窗口冲突场景

### 6. Lifecycle and Cleanup

- 数据是否可复用
- 是否需要清理
- 是否会污染环境
- 是否会影响其他测试任务

### 7. Observability

- 是否便于通过页面 / 接口 / 数据库观察结果
- 是否具备唯一标识便于追踪
- 是否便于定位该批测试数据

## 🧪 Design Strategies by Scenario

### 1. For Functional Testing

重点关注：

- 数据是否覆盖主流程
- 是否具备关键状态
- 是否能验证失败路径和边界条件

### 2. For API Testing

重点关注：

- 前置依赖对象是否完整
- 请求参数引用的数据是否存在
- 数据状态是否能触发不同接口结果
- 是否需要接口前后数据校验

### 3. For UI Testing

重点关注：

- 页面是否能展示不同数据状态
- 是否能触发空态、加载态、异常态
- 不同角色下是否有不同展示数据

### 4. For Regression Testing

重点关注：

- 是否有稳定的基础数据集
- 是否能快速重建
- 是否能减少环境随机因素带来的波动

### 5. For Automated Testing

重点关注：

- 数据是否可脚本化创建
- 是否能独立执行
- 是否能重置和清理
- 是否适合并行执行
- 是否避免相互污染

### 6. For Defect Reproduction

重点关注：

- 是否能构造触发问题的特定状态
- 是否能还原历史脏数据、边界数据、冲突数据
- 是否能支持修复前后对比验证

最终输出时，应尽量包含以下内容：

- 测试目标

- 每个对象所需状态
- 数据构建方式
- 数据唯一标识建议
- 复用策略
- 清理 / 重置策略
- 风险与限制

## 🚫 Common Mistakes

构建测试数据时，必须避免以下问题：

- 只造主流程数据，不造异常和边界数据
- 只关注字段值，不关注对象状态
- 数据能创建，但无法支持完整流程
- 多个测试任务共用同一批易污染数据
- 不考虑数据回收和环境清理
- 只靠手工随机造数，无法复现
- 数据与真实业务规则不一致
- 不保留唯一标识，导致测试后无法追踪

## ✅ Good Output Characteristics

高质量测试数据设计通常具备以下特征：

- 测试目标清晰

- 可复用性清晰
- 清理策略清晰

> 让测试数据构建具备场景意识、状态意识、隔离意识和复用意识，能够稳定支撑功能测试、接口测试、自动化测试、回归测试和异常验证，并沉淀为长期可复用的测试资产。
