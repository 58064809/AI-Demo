# 🛠️ Skill Name

接口测试技能

---

## 🎯 Purpose

本技能用于定义接口测试分析、接口测试设计和接口验证的具体处理方法，帮助 Agent 在面对接口文档、接口规则、联调场景、自动化输入整理和接口风险评估时，能够以结构化、系统化、可复用的方式完成高质量接口测试工作。

它解决的问题不是“把参数列出来”，而是：

- 如何把接口规则转成测试点
- 如何系统覆盖参数、鉴权、状态、数据、异常和幂等等维度
- 如何识别接口隐藏风险
- 如何为接口自动化和回归提供高质量输入
- 如何避免只测“通不通”，而漏掉“对不对”

---

## 🧠 Core Principles

### 1. Contract Before Validation

先理解接口契约，再设计验证内容。  
接口测试不是直接对着参数表写几个用例，而是先确认接口定义、字段含义、前置条件和行为规则。

### 2. Rule Before Response Code

不能把“返回 200”直接等同于“接口正确”。  
必须验证接口行为、业务结果、状态变化和数据变化。

### 3. Input-Output-State Integration

接口测试不仅看输入和输出，还要看调用前后状态是否正确变化、数据是否正确落地、上下游是否正确联动。

### 4. Exception Sensitivity

要对非法输入、边界输入、鉴权失败、幂等失效、重复提交、重试冲突、依赖失败等场景保持高敏感度。

### 5. Automation-Friendly Design

接口测试输出应尽量适合作为自动化测试输入，包括请求样例、断言点、前置依赖、数据准备和清理思路。

---

## 📦 Applicable Scenarios

本技能适用于但不限于以下场景：

- 根据接口文档设计测试点
- 根据 Swagger / OpenAPI 设计测试用例
- 联调前补接口验证清单
- 接口自动化前整理断言点和依赖关系
- 识别接口幂等、鉴权、并发、状态流转风险
- 根据线上问题补接口侧测试覆盖
- 评估版本改动对接口的回归影响

---

## 🔍 Input Requirements

开始接口测试前，尽量先明确以下输入：

- 接口路径是什么
- 请求方法是什么
- 输入参数有哪些
- 输出字段有哪些
- 字段类型、约束、枚举和默认值是否明确
- 接口适用于哪些角色
- 调用前需要满足哪些状态
- 调用后应该发生哪些状态变化和数据变化
- 是否存在幂等、重试、并发、限流、超时等特殊规则
- 是否依赖外部系统或异步任务

如果上述信息不完整，应先列出前提和待确认项，再继续设计。

## 🧩 Standard Design Process

### Step 1：Clarify the Interface Purpose

先明确接口的核心用途：

- 查询类接口
- 创建类接口
- 更新类接口
- 删除类接口
- 状态变更类接口
- 触发动作类接口
- 异步任务触发类接口

不同类型接口，测试重点不同。

---

### Step 2：Read the Contract Carefully

梳理接口契约，包括：

- 路径
- 方法
- Header
- Query 参数
- Path 参数
- Body 参数
- 返回结构
- 状态码
- 错误码
- 字段语义
- 取值范围
- 是否必填
- 是否存在字段依赖关系

如果文档存在缺失、冲突或语义不清，必须显式记录。

---

### Step 3：Extract Business Rules

梳理接口业务规则，包括：

- 什么条件下允许调用
- 什么条件下必须拒绝
- 哪些角色能调
- 哪些状态能调
- 成功后状态如何变化
- 失败后如何反馈
- 是否允许重复调用
- 是否必须幂等
- 是否受次数、时间、金额、数量等限制

---

### Step 4：Split into Test Dimensions

默认从以下维度拆分：

- 参数校验
- 鉴权校验
- 权限校验
- 业务规则校验
- 状态流转校验
- 返回结果校验
- 错误码校验
- 数据一致性校验
- 幂等校验
- 重复提交 / 并发 / 重试校验
- 外部依赖 / 异步链路校验（如适用）

---

### Step 5：Design Assertions

接口测试不仅要有请求，还要有断言。  
断言应尽量覆盖：

- 状态码
- 错误码
- 返回字段
- 返回字段值
- 状态变化
- 数据库变化
- 缓存变化
- 异步结果
- 下游联动结果

---

### Step 6：Review for Gaps and Hidden Risks

完成初稿后，至少检查以下问题：

#### 是否漏掉异常和边界

例如：

- 缺字段
- 空值
- 类型错误
- 非法枚举
- 边界长度
- 非法状态调用
- 越权调用

#### 是否漏掉行为类风险

例如：

- 幂等失效
- 重复下单
- 重复发放
- 重试放大
- 并发冲突
- 状态机错乱
- 缓存 / 数据库不一致

#### 是否只验证了“返回”，没有验证“结果”

例如接口返回成功，但状态没变、数据没写、异步没成功。

---

## 📌 Default Coverage Checklist

设计接口测试时，默认优先检查以下覆盖项：

### 1. Basic Contract

- 路径是否正确
- 请求方法是否正确
- 参数位置是否正确
- 返回结构是否正确
- 字段类型是否符合定义

### 2. Required / Optional Fields

- 必填字段缺失如何处理
- 非必填字段缺失是否有默认行为
- 字段依赖关系是否正确处理

### 3. Field Boundary and Invalid Input

- 长度边界
- 数值边界
- 枚举非法值
- 类型错误
- 格式错误
- 特殊字符
- 空字符串 / Null / 空数组
- 超长值 / 超大值 / 精度问题

### 4. Authentication and Authorization

- 未登录调用如何处理
- Token 缺失 / 失效 / 错误如何处理
- 不同角色是否返回不同结果
- 是否存在越权调用风险

### 5. Business Rule Validation

- 状态不满足时是否拒绝
- 条件不满足时是否拒绝
- 是否存在绕过前端直接调用成功的问题
- 是否正确校验前置依赖

### 6. Response Validation

- 成功场景状态码是否合理
- 失败场景状态码是否合理
- 错误码是否准确
- 返回字段是否齐全
- 返回字段值是否正确
- 是否存在无意义或误导性返回

### 7. State Transition

- 调用前状态是否正确
- 调用后状态是否正确
- 是否允许重复调用
- 是否允许逆向调用
- 非法状态调用是否被拦截

### 8. Data Consistency

- 数据库是否正确落库
- 缓存是否同步更新
- 返回结果与实际状态是否一致
- 是否存在重复写入、漏写、脏数据

### 9. Repeat / Retry / Idempotency

- 重复请求是否重复创建资源
- 重试是否造成副作用
- 幂等接口是否真正幂等
- 网络抖动下重复请求是否安全

### 10. Concurrency and Dependency

- 并发请求是否会冲突
- 外部依赖失败如何处理
- 异步任务失败或延迟如何体现
- 链路超时是否有明确行为

## 🧪 Design Strategies by Interface Type

### 1. Query Interface

重点关注：

- 查询条件组合
- 空结果
- 大结果集
- 分页
- 排序
- 过滤条件边界
- 权限过滤
- 展示字段准确性

### 2. Create Interface

重点关注：

- 必填校验
- 重复创建
- 唯一性约束
- 成功后资源创建结果
- 状态初始值
- 幂等和重复提交问题

### 3. Update Interface

重点关注：

- 更新字段约束
- 非法状态更新
- 局部更新与全量更新差异
- 并发更新冲突
- 更新后数据一致性

### 4. Delete Interface

重点关注：

- 删除权限
- 删除前置条件
- 物理删除 / 逻辑删除差异
- 已删除对象重复删除
- 关联数据处理

### 5. State Transition Interface

重点关注：

- 合法状态流转
- 非法状态流转
- 重复触发
- 逆向流转
- 状态变更后的联动动作

### 6. Action Trigger Interface

重点关注：

- 动作是否允许重复触发
- 是否具备幂等保护
- 是否触发异步任务
- 成功返回与实际动作完成是否一致

## 🚫 Common Mistakes

接口测试设计时，必须避免以下问题：

- 只验证 200，不验证业务结果
- 只看文档，不看业务规则
- 只写参数，不写状态和数据变化
- 只写成功场景，不写失败和异常场景
- 忽略鉴权、权限、幂等、重试和并发问题
- 文档不清却默认接口定义正确
- 接口自动化脚本先行，测试设计滞后
- 返回成功但不验证数据库 / 缓存 / 异步结果

---

## ✅ Good Output Characteristics

高质量接口测试设计通常具备以下特征：

- 契约理解清晰
- 业务规则拆解清晰
- 参数与状态覆盖完整
- 异常和边界覆盖充分
- 风险点明确
- 断言点明确
- 可直接转为自动化输入
- 可直接支持联调、回归和上线判断
